## Problem Statement

现有设计文档定义了教学智能体的核心功能：用户扮演教师，配置学生群体，选择教学模式（灌输式、启发式、讨论式），进行教学活动。

**新增需求**：添加一种**观察模式**（Observation Mode），用户不参与教学，而是作为隐形观察者观看教师 agent 和学生 agent 的自动互动，并在会话结束后获得量化分析报告。这是为了满足教育研究人员的场景：研究不同教学模式对教学效果的影响。

## What Makes This Cool

核心亮点是**教学模式效果的量化可见性**。教育研究人员可以：
- 配置相同的课程和学生群体，切换不同教学模式，对比观察效果差异
- 获得量化分析报告：学生参与度、回答正确率、互动频率、知识点掌握度等
- 可重复、可控的教学实验环境

这不仅是"看 agent 聊天"，而是"研究教学法"的工具。

## Constraints

- 技术栈已选定：FastAPI + LangChain + React
- 学习/黑客松项目，非创业方向
- 注重深度优先，单一功能做到位
- 使用硅基流动免费模型（Qwen2.5-7B-Instruct）
- 必须与现有设计兼容（教学模式、学生群体、消息流）

## Premises

1. **用户需要「观察」而非「控制」** — 观察模式用户不参与教学，只看 agent 自动互动
2. **分析报告需要量化数据** — 结束后生成的报告包含可量化指标（参与度、正确率、互动频率等）
3. **与核心教学模式兼容** — 观察模式适用于所有三种教学模式（灌输式、启发式、讨论式）

## Approaches Considered

### Approach A: Minimal — 被动观察模式

只读方式观看预设配置的会话，结束后显示基本统计。

- **Effort**: S (最小)
- **Risk**: Low
- **Pros**: 实现最快，不修改现有逻辑
- **Cons**: 无法控制配置，灵活性有限，研究价值低

### Approach B: Balanced — 可配置观察模式（已选择）

用户配置课程/模式/学生群体，实时观看，支持暂停/继续，生成详细分析报告。

- **Effort**: M (中等)
- **Risk**: Medium
- **Pros**: 完全匹配研究需求，支持控制变量和对比研究，与现有设计集成
- **Cons**: 需要新增 UI 和分析逻辑

### Approach C: Enhanced — 实验室模式

增加实时状态可视化、多会话对比、数据导出。

- **Effort**: L (大)
- **Risk**: High
- **Pros**: 最强大的研究工具
- **Cons**: 实现复杂度高，适合作为未来扩展

## Recommended Approach

**Approach B: Balanced — 可配置观察模式**

原因：
1. 完全匹配教育研究人员的实际需求
2. 支持控制变量研究（相同课程/学生，不同教学模式）
3. 量化分析报告提供研究级数据
4. 与现有设计（教学模式、学生群体）无缝集成
5. 实现难度适中，可作为独立功能模块开发

## Observation Mode Architecture

### User Flow

```
1. 用户进入应用，选择「观察模式」标签
   ↓
2. 配置观察会话：
   - 选择课程（二次函数、二叉树遍历、牛顿第二定律）
   - 选择教学模式（灌输式/启发式/讨论式）
   - 配置学生群体（使用现有的 StudentGroupConfig 机制）
   ↓
3. 点击「开始观察」
   ↓
4. 系统创建只读会话，agent 自动互动
   - 前端实时显示对话（只读）
   - 用户可暂停/继续（不影响 agent 逻辑，只是暂停前端显示）
   - 显示当前会话状态：当前模式、已进行时间、消息数量
   ↓
5. 会话结束（达到时长或教学完成）
   ↓
6. 生成并展示分析报告
```

### Session State Extension

在现有 `TeachingSession` 基础上扩展：

```python
class TeachingSession:
    # ... 现有字段 ...

    # 新增：观察模式相关
    is_observation_mode: bool = False  # 是否为观察模式
    observer_read_only: bool = True   # 观察者只读标记

class ObservationMetrics:
    """观察模式分析指标"""
    session_id: str

    # 教学模式
    teaching_mode: TeachingMode

    # 课程信息
    course_id: str
    topic: str

    # 时间指标
    duration_seconds: int
    message_count: int
    teacher_message_count: int
    student_message_count: int

    # 互动指标
    interaction_frequency: float  # 每分钟互动次数
    student_participation_rate: float  # 学生参与率（主动发言/总发言）

    # 学习效果指标
    average_knowledge_gain: float  # 平均知识掌握度提升
    average_correct_rate: float    # 平均回答正确率

    # 学生群体维度统计
    group_metrics: Dict[str, GroupMetrics]

class GroupMetrics:
    """学生群体维度统计"""
    group_id: str
    group_name: str

    # 该群体的平均指标
    avg_level: StudentLevel
    avg_attitude: StudentAttitude
    avg_knowledge_gain: float
    avg_participation: float
    avg_correct_rate: float
```

## Success Criteria

**可观测的成功标准**:
1. **观察模式可进入**: 用户能在 3 步内进入观察模式（选模式 → 配置 → 开始）
2. **实时对话可见**: 前端实时显示 agent 对话，流畅无延迟
3. **只读性保证**: 观察者无法干预 agent 行为（无输入框、无控制按钮）
4. **分析报告量化**: 结束后显示至少 5 个量化指标，支持群体维度对比
5. **可演示**: 2 分钟内完成：进入观察模式 → 观看 1-2 轮对话 → 查看报告

## Open Questions

1. 观察模式下，agent 是否需要加速运行？（建议：正常速度，便于观察细节）
2. 分析报告是否需要保存历史记录？（建议：v1 仅显示当次报告，未来支持历史对比）

## What I noticed about how you think

- 你说"观察教师 agent 和学生 agent 的互动方式" —— 这说明你关注的是**agent 协作的过程**，而不仅仅是结果。这是研究思维，不是演示思维。
- 你选择"教育研究人员"作为目标受众 —— 说明你在意这个功能的**实用价值**，而不只是"看起来很酷"。这是 builder 的成熟信号。
- 你选择"教学模式研究"作为差异化 —— 这抓住了现有设计的核心价值（三种教学模式），并将观察模式与之深度集成，而不是做一个独立的"看聊天"功能。
