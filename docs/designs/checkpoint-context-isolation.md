# Design: 检查点级上下文隔离 — 解决教师 Agent 上下文溢出

## Problem Statement

教师 Agent 的 LLM 上下文在课堂执行过程中无限制增长。每条消息都追加到 `SessionMemory.message_history`，到课堂后期（尤其是最终总结 `end_feedback()` 时），上下文超出 7B 模型的 32K 窗口限制，导致调用失败。

两个具体痛点：
1. **检查点间上下文累积**: 所有检查点的消息堆积在同一个 `message_history` 中，教师无法"忘记"早期内容
2. **最终总结溢出**: `end_feedback()` 使用完整的 `_build_system_prompt()`（含 50 条历史消息），经历一整堂课后必然超限

## What Makes This Cool

这不仅仅是"加个截断"。它映射了真实教师的教学认知模型：每个知识点讲完后，教师在脑中做一个简短的回顾总结，然后带着这些要点进入下一个知识点。检查点边界就是自然的认知分段点。

## Constraints

- 使用 Qwen2.5-7B-Instruct（32K 上下文窗口），不是大模型，需要严格控制 prompt 长度
- 不能丢失跨检查点的结构化状态（已讲授主题、学生参与度、误解记录）—— `TeacherAgentMemory` 已有这些
- 学生 Agent 不受影响（已有 10 条消息限制）
- 数据库持久化不受影响（`MessageService.emit_message()` 在重置前已持久化）
- 需要同时修复观察模式（`SessionOrchestrator`）和教师模式（`TeacherSessionController`）

## Premises

1. **检查点摘要 + TeacherAgentMemory = 足够的跨检查点上下文。** 结构化记忆（covered_topics, misconceptions, participation）自然跨检查点保留。LLM 生成的摘要捕获叙事上下文（讨论了什么、学生如何回应）。两者结合为下一个检查点提供充分上下文，无需原始消息历史。
2. **最终总结应从累积的检查点摘要构建，而非原始消息。** 这是解决溢出问题的核心。用 N 个检查点摘要 + TeacherAgentMemory 状态替代 50+ 条消息。
3. **在检查点边界清除 `message_history` 是安全的。** TeacherAgent 仅通过 `get_agent_context()` 读取历史消息，不会直接索引 `message_history`。编排器通过 `MessageService.emit_message()` 持久化消息，该操作在重置前完成。注意：`TeacherSessionController` 直接调用 `message_history.append()`（9 处），但重置在检查点推进时显式触发，不影响持久化。
4. **检查点摘要应由 LLM 生成，而非规则提取。** 规则摘要（"3 个问题，5 名学生回应"）丢失语义内容。LLM 可将检查点消息压缩为保留教学叙事的简洁段落。
5. **每个检查点额外一次 LLM 调用（摘要生成）的成本可接受。** 摘要 prompt 输入短（仅当前检查点消息），输出短（一段话），延迟低。

## Approaches Considered

### Approach A: 检查点重置 + LLM 摘要（最小方案）
每个检查点完成后，LLM 将消息压缩为短摘要存入 `checkpoint_summaries` 列表。清除 `message_history`。`end_feedback()` 读取摘要而非原始消息。

### Approach B: 滑动窗口（无检查点边界）
直接将 `message_history` 上限设为 N 条（类似学生 Agent 的 10 条）。启用现有 `summary_fn` 维护 `teaching_summary`。

### Approach C: 分层上下文（丰富的检查点状态）
每个检查点维护完整状态对象：{title, summary, key_messages, misconceptions_found, participation_snapshot}。`get_agent_context()` 组装多检查点视图。

## Recommended Approach

**Approach A: 检查点重置 + LLM 摘要。**

原因：
- 同时解决两个问题（检查点间累积 + 最终总结溢出）
- 代码改动最小：修改 ~3 个文件，~100 行
- 符合用户 stated goal
- 每检查点一次额外 LLM 调用，成本可控

### 具体改动

#### 1. SessionMemory 新增 `checkpoint_summaries` 字段

```python
# session_memory.py
@dataclass
class SessionMemory:
    session_id: int
    topic: str
    message_history: list[Message] = field(default_factory=list)
    teaching_summary: str = ""
    checkpoint_summaries: list[str] = field(default_factory=list)  # 新增

    # 新增方法
    def clear_message_history(self) -> None:
        """清除消息历史（检查点边界调用）.

        同时将 last_summary_update 设为 0，因为新的检查点
        消息计数从 0 开始。
        """
        self.message_history.clear()
        self.last_summary_update = 0

    def get_full_context(self) -> str:
        """获取完整上下文（包含所有检查点摘要）.

        用于最终总结等需要全局视角的场景。
        包含 teaching_summary（如果有）和所有 checkpoint_summaries。
        """
        parts = [f"教学主题: {self.topic}"]
        if self.teaching_summary:
            parts.append(f"教学摘要: {self.teaching_summary}")
        if self.checkpoint_summaries:
            parts.append("各检查点教学摘要:")
            for i, summary in enumerate(self.checkpoint_summaries, 1):
                parts.append(f"  检查点{i}: {summary}")
        parts.append("最近的对话:")
        for msg in self.message_history[-self.max_history_messages:]:
            parts.append(f"[{msg.sender}] ({msg.message_type.value}): {msg.content}")
        return "\n".join(parts)
```

#### 2. MemoryManager 新增 `summarize_checkpoint()` 方法

```python
# memory_manager.py
def summarize_checkpoint(self) -> str | None:
    """生成当前检查点的摘要并重置消息历史.

    Returns:
        生成的检查点摘要，如果无法生成则返回 None
    """
    messages = self.session_memory.message_history
    if not messages:
        return None

    if self.summary_fn is None:
        return None

    # 构建摘要 prompt
    prompt = (
        "请将以下课堂对话摘要为一段简洁的教学总结（100字以内）。\n"
        "重点包括：讲授了什么知识点、学生理解情况、发现的误解。\n\n"
        + "\n".join(
            f"[{m.sender}]: {m.content}" for m in messages
        )
    )

    summary = self.summary_fn(prompt)
    if summary:
        self.session_memory.checkpoint_summaries.append(summary)

    # 无论摘要是否成功生成，都清除消息历史
    # 摘要失败时只丢失叙事上下文，TeacherAgentMemory 的结构化状态不受影响
    self.session_memory.clear_message_history()

    return summary
```

#### 4. TeacherAgent.end_feedback() 使用 `get_full_context()`

```python
# teacher_agent.py
def _build_end_feedback_context(self) -> str:
    """构建最终总结的轻量级上下文.

    使用累积的检查点摘要 + 教师记忆，而非原始消息历史。
    注意：此方法故意不包含教学模式指令（_MODE_INSTRUCTIONS），
    因为最终总结不需要区分灌输式/启发式/讨论式。

## 基本信息
- 教学主题: {topic}

{teacher_context}

{full_context}
"""

def end_feedback(self) -> str:
    system_prompt = self._build_end_feedback_context()
```

#### 5. 注入 summary_fn

在 `SessionOrchestrator` 和 `TeacherSessionController` 的初始化中，将 LLM 调用作为 `summary_fn` 传入 `MemoryManager`。

`llm` 参数来源：与 `TeacherAgent` 共享同一个 LLM 实例（`teacher_agent.llm`），或在初始化时单独传入。

```python
# 在创建 MemoryManager 时注入 summary_fn
from core.llm_utils import safe_llm_call

def _make_summary_fn(llm):
    """创建检查点摘要函数.

    Args:
        llm: 与 TeacherAgent 共享的 LLM 实例
    """
    def summary_fn(prompt: str) -> str:
        messages = [
            {"role": "system", "content": "你是一个教学摘要助手。"},
            {"role": "user", "content": prompt},
        ]
        return safe_llm_call(llm.invoke, "摘要", "checkpoint", messages, temperature=0.3)
    return summary_fn

# 使用示例
summary_fn = _make_summary_fn(teacher_agent.llm)
memory_manager = MemoryManager(session_memory=session_memory, summary_fn=summary_fn)
```

### 设计决策：正常教学中不包含历史检查点摘要

这是一个有意的权衡：在检查点 N+1 的正常教学（讲授、提问、回复）中，教师 Agent **只能看到**：
- `TeacherAgentMemory` 的结构化状态（covered_topics, misconceptions, participation）
- 当前检查点的消息历史（刚刚清除过，所以只有本检查点的消息）
- 系统提示词中的模式指令

教师**看不到**之前检查点的摘要。原因：
1. 摘要只在 `end_feedback()` 中使用（`get_full_context()`），这是唯一需要全局视角的场景
2. `TeacherAgentMemory` 已经携带了跨检查点的关键信息（讲过什么、学生表现如何）
3. 将历史摘要塞入每次 LLM 调用会增加不必要的 token 消耗，对 7B 模型的 32K 窗口不友好

如果后续发现教师讲授质量下降（无法引用之前的知识点），可以将**最后一个**检查点的摘要注入 `get_agent_context()`，而不是全部。

### 上下文大小估算

| 场景 | 改动前 | 改动后 |
|------|--------|--------|
| 检查点讲授 | 50 条消息 + system prompt (~8K tokens) | 当前检查点消息 + system prompt (~3K tokens) |
| 检查点提问 | 50 条消息 + system prompt (~8K tokens) | 当前检查点消息 + system prompt (~3K tokens) |
| 最终总结 | 50 条消息 + 教学记忆 + system prompt (~10K tokens) | N 个检查点摘要 (~1-2K tokens) + 教学记忆 + system prompt (~3K tokens) |
| 作业布置 | 50 条消息 + system prompt (~8K tokens) | 最后一个检查点的消息 + system prompt (~3K tokens) |

注：作业布置在最后一个检查点重置后执行，此时 `message_history` 只有本检查点消息，所以 `_build_system_prompt()` 自然安全。

### TeacherSessionController 的特殊处理

`TeacherSessionController` 有 9 处直接调用 `session_memory.message_history.append()`，绕过了 `MemoryManager.process_message()`。因此：
- 不能依赖 `process_message()` 内部的自动摘要触发
- 必须在 `handle_advance_checkpoint()` 中**显式调用** `memory_manager.summarize_checkpoint()`
- 调用时机：在 WebSocket 广播检查点状态变更之后、返回响应之前

### teaching_summary 与 checkpoint_summaries 的关系

两个字段服务于不同目的，设计上互不干扰：

- **`teaching_summary`**（滚动摘要）：由 `MemoryManager._check_and_update_summary()` 维护，每 10 条消息更新一次。`clear_message_history()` 将 `last_summary_update` 重置为 0，使其在每个新检查点重新开始累积。它反映的是**当前检查点**的教学摘要。

- **`checkpoint_summaries`**（累积摘要）：由 `summarize_checkpoint()` 在每个检查点完成时生成，追加到列表中，跨检查点累积。它反映的是**所有已完成检查点**的教学摘要。

两者不冲突：`teaching_summary` 在当前检查点内滚动更新，`checkpoint_summaries` 在检查点边界一次性生成。

## Success Criteria

- 5 个检查点 + 5 个学生的完整课堂能顺利执行到最终总结，不出现上下文溢出
- 最终总结内容能覆盖所有检查点的核心知识点（通过人工审查）
- 每个检查点的教师讲授质量不下降（对比改动前后的讲授内容）
- 现有 245 个测试全部通过
