## Problem Statement

后端当前使用 SQLite + aiosqlite 作为数据库，需要迁移到 PostgreSQL + asyncpg。项目是学习/研究性质的教学智能体系统，迁移目标是让数据库层支持生产级特性（并发、JSONB、ENUM 类型），同时保持所有业务逻辑不变。

## What Makes This Cool

这不是一个无聊的"换个数据库"任务。迁移到 PostgreSQL 后，原本因为 SQLite 限制而做的 workaround（JSON 存为 TEXT、手动 PRAGMA foreign_keys）都可以用 PostgreSQL 原生特性替代。更关键的是，项目中已知的 JSON 并发修改竞态问题（`async-race-condition-json-mutation` learning）可以用 PostgreSQL 的 `jsonb_set()` 原子操作优化。

## Constraints

- 只改数据库层，业务逻辑（agent、service、websocket handler）不动
- SQLAlchemy ORM 层已经做了良好的抽象，大部分代码不需要改动
- 6 个 ORM 模型，4 个现有 Alembic 迁移（全部删除，重新生成基线）
- 测试需要同时适配：单元/集成测试用 SQLite 内存数据库（快速），可选 PostgreSQL 集成测试

## Premises

1. **全新基线迁移** — 删除现有 4 个 Alembic 迁移，生成一个包含所有表的 PostgreSQL 原生迁移。迁移历史不重要，干净比兼容重要。
2. **asyncpg 作为异步驱动** — 标准选择，SQLAlchemy 2.x 对 asyncpg 支持最好。
3. **测试继续用 SQLite** — 单元测试和大部分集成测试继续用 SQLite 内存数据库（快速），只有需要验证 PostgreSQL 特定行为的测试才连接真实 PostgreSQL。ORM 模型使用 SQLAlchemy 通用类型（`JSON`、`Enum`）而非 PostgreSQL 特定类型（`JSONB`、`postgresql.ENUM`），这样同一个模型定义可以同时兼容 SQLite 和 PostgreSQL。
4. **环境变量管理数据库连接** — DATABASE_URL 通过 .env 配置，不再硬编码在 YAML 中。`settings.py` 负责加载（env var 优先，YAML 兜底）。

## Approaches Considered

### Approach A: 全新基线 + 最小改动

删除所有现有迁移，生成一个 PostgreSQL 原生类型的全新基线迁移。改动范围：database.py、alembic 配置、requirements.txt、conftest.py。ORM 模型类型定义不做调整，依赖 SQLAlchemy 的自动类型映射。

- Effort: S
- Risk: Low
- Pros: 最简单，改动最少，风险最低，ORM 模型零改动
- Cons: 不利用 PostgreSQL 特定优化（JSONB 原子操作）
- Reuses: 现有 ORM 模型、Alembic 配置结构、conftest.py 框架

### Approach B: 全新基线 + Schema 增强

在 Approach A 基础上，趁迁移机会改进 Schema：
- 保持 ORM 使用 SQLAlchemy 通用类型（`JSON`、`Enum`）确保测试兼容
- 利用 PostgreSQL 的自动映射：`JSON` → `JSONB`，`Enum` → 原生 `ENUM`
- 用 `jsonb_set()` 优化已知的 JSON 竞态问题（`persistence_service.py` 已有 `with_for_update()` 行锁保护，`jsonb_set()` 是额外优化而非 bug 修复）

- Effort: M
- Risk: Low
- Pros: 利用 PostgreSQL 特性优化，Schema 更规范
- Cons: 改动范围稍大
- Reuses: 同 Approach A，加上对已有 race condition learning 的应用

### Approach C: 双数据库支持（SQLite + PostgreSQL）

通过抽象层同时支持 SQLite 和 PostgreSQL，本地开发用 SQLite，部署用 PostgreSQL。

- Effort: L
- Risk: Med
- Pros: 开发体验好，不强制要求本地安装 PostgreSQL
- Cons: 过度工程化（学习项目），需要维护两套 Schema 兼容性，容易踩坑
- Reuses: 现有代码结构

## Recommended Approach

**Approach B: 全新基线 + Schema 增强** — ORM 模型保持使用 SQLAlchemy 通用类型（确保 SQLite 测试兼容），PostgreSQL 自动将其映射为原生类型。额外用 `jsonb_set()` 优化 JSON 操作。对于一个学习项目来说，这正是学习 PostgreSQL 特性的好机会。

不选 A 是因为错过了学习 PostgreSQL 特性的机会。不选 C 是因为学习项目不需要双数据库支持，SQLite 内存数据库做测试就够了。

## 类型兼容策略

**核心问题**：ORM 模型需要同时兼容 SQLite（测试）和 PostgreSQL（生产）。

**解决方案**：使用 SQLAlchemy 通用类型，不做 PostgreSQL 特定导入：

```python
# ✅ 正确 — 通用类型，两种数据库都支持
sa.Column("plan_data", sa.JSON)           # SQLite → TEXT, PostgreSQL → JSONB
sa.Column("level", sa.Enum(StudentLevel)) # SQLite → VARCHAR CHECK, PostgreSQL → ENUM

# ❌ 错误 — PostgreSQL 特定类型，SQLite 测试会报错
from sqlalchemy.dialects.postgresql import JSONB
sa.Column("plan_data", JSONB)
```

SQLAlchemy 2.x 的 `sa.JSON` 在 PostgreSQL 上自动使用 `JSONB`，在 SQLite 上使用 `TEXT`。`sa.Enum` 同理，自动适配。不需要 `TypeDecorator` 或 `with_variant()`。

## 关键代码变更示例

### settings.py（添加数据库配置加载）

```python

    "DATABASE_URL",
    _DATABASE["database"]["url"],
)

# 连接池配置
DATABASE_POOL_SIZE: int = _DATABASE["database"].get("pool_size", 5)
DATABASE_MAX_OVERFLOW: int = _DATABASE["database"].get("max_overflow", 10)
```

### database.py（引擎配置）

```python
# Before (SQLite)
ASYNC_DATABASE_URL = "sqlite+aiosqlite:///datas/database.db"
async_engine = create_async_engine(
    ASYNC_DATABASE_URL,
    connect_args={"check_same_thread": False},
)

# After (PostgreSQL)
from core.settings import DATABASE_URL, DATABASE_POOL_SIZE, DATABASE_MAX_OVERFLOW

async_engine = create_async_engine(
    DATABASE_URL,  # postgresql+asyncpg://admin:123456@localhost:5432/mydb
    pool_size=DATABASE_POOL_SIZE,       # 5
    max_overflow=DATABASE_MAX_OVERFLOW, # 10
    pool_pre_ping=True,                 # 防止连接 stale
    echo=False,
)
```

"""Alembic 迁移测试 - 验证数据库表结构正确创建."""

from sqlalchemy import inspect

class TestAlembicMigration:
    """使用 SQLAlchemy inspect() 检查表结构（数据库无关）."""

    async def test_all_tables_exist(self, db_session):
        """测试所有表是否已创建."""
        from sqlalchemy import text
        result = await db_session.execute(text(
            "SELECT table_name FROM information_schema.tables WHERE table_schema='public'"
        ))
        # 或者对 SQLite: SELECT name FROM sqlite_master WHERE type='table'
        tables = {row[0] for row in result.fetchall()}
        assert "teaching_sessions" in tables
        assert "messages" in tables
        assert "session_memories" in tables
        assert "teacher_memories" in tables
        assert "student_memories" in tables
        assert "checkpoint_plans" in tables

    async def test_columns_exist(self, db_session):
        """测试关键列存在（不检查类型名，因为 SQLite 和 PostgreSQL 类型名不同）."""
        inspector = inspect(db_session.bind)
        columns = {c["name"] for c in inspector.get_columns("teaching_sessions")}
        assert "students_config" in columns
        assert "teaching_mode" in columns
        assert "topic" in columns

    async def test_foreign_keys_exist(self, db_session):
        """测试外键约束存在."""
        inspector = inspect(db_session.bind)
        fks = inspector.get_foreign_keys("messages")
        fk_refs = {(fk["referred_table"], fk["constrained_columns"]) for fk in fks}
        assert ("teaching_sessions", ["session_id"]) in fk_refs

## PostgreSQL 本地环境

用户本地已通过 Docker 配置好 PostgreSQL，连接信息通过 `.env` 管理：

```yaml
# docker-compose.yml
services:
  postgres:
    image: postgres:16
    environment:
      POSTGRES_USER: admin
      POSTGRES_PASSWORD: 123456
      POSTGRES_DB: mydb
    ports:
      - "5432:5432"
```

```
# backend/.env
DATABASE_URL=postgresql+asyncpg://admin:123456@localhost:5432/mydb
```

Alembic 迁移 URL（同步驱动）：
```
sqlalchemy.url = postgresql+psycopg2://admin:123456@localhost:5432/mydb
```

可选 PostgreSQL 集成测试：
```
TEST_POSTGRESQL_URL=postgresql+asyncpg://admin:123456@localhost:5432/mydb
```
